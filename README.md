## What It Does

Users upload PDF or text documents via an API. The system ingests them — extracting text, splitting into chunks, computing semantic embeddings, and storing them in a vector index. When a user asks a question, the system retrieves the most relevant chunks using vector similarity search, constructs a grounded prompt, and generates an answer using a large language model. Sources are returned alongside the answer for verification.

Three API endpoints, each with a clear contract defined in `specs/openapi.yaml`:

| Endpoint | Purpose |
|----------|---------|
| `POST /upload` | Ingest a document into the vector index |
| `POST /query` | Ask a question, get an answer with sources |
| `GET /health` | Service liveness check |

## The Development Philosophy

### No Handwritten Code

This project was built through a single extended conversation with Claude Code. The workflow was:

1. State the architecture and requirements in plain English
2. Claude generates the code, specs, config, and project structure
3. Review, refine, iterate — all in natural language

No editor was opened to write code. No syntax was typed by hand. The human role was **architect and reviewer**: deciding what to build, validating the output, and steering when the direction was wrong.

This is not a gimmick. It reflects a genuine shift in how software can be made. The bottleneck moves from *how to write code* to *how to think clearly about what you want*. The specs, the architecture decisions, the choice of abstractions — those are the hard parts. The code is the output of having thought clearly.

### Spec-Driven Development

Code was not written until contracts were defined. Every `specs/` file was created before any implementation:

```
specs/
├── openapi.yaml      ← API contract: endpoints, schemas, status codes
├── domain.yaml       ← Domain model: Document, Chunk, Query, RetrievalResult
├── rag-pipeline.md   ← Pipeline flows: ingestion and retrieval/generation
└── prompts.yaml      ← Prompt text, generation parameters, constraints
```

Specs serve as the **source of truth**. When behaviour needs to change, the spec changes first, and the implementation follows. This has several properties that matter:

- **Contracts are explicit before code exists.** There is no ambiguity about what a `RetrievalResult` contains or what status codes `/upload` returns.
- **Multiple language implementations stay in sync.** The TypeScript types in `packages/shared/src/types/index.ts` and the Python dataclasses in `services/rag-service/retrieval/retriever.py` are both derived from `specs/domain.yaml`.
- **The AI has a reference to validate against.** Claude Code can check whether generated code matches the spec, rather than inventing behaviour arbitrarily.
- **Prompts are versioned like code.** `specs/prompts.yaml` contains the exact system prompt, context format, and generation constraints. Changing a prompt is a tracked, deliberate decision — not an ad-hoc edit buried in application code.

## Architecture

### Polyglot Design: Why Two Languages

The system is intentionally split between two languages, each used where it has a clear advantage:

**TypeScript** owns the **API and orchestration layer**:
- Lambda handler boilerplate and routing
- Request validation and error mapping
- AWS SDK calls (Lambda invoke, API Gateway integration)
- Infrastructure as code (AWS CDK)

TypeScript excels here because the AWS ecosystem is deeply TypeScript-native, CDK is a first-class TypeScript library, and the handler logic is primarily glue code with no algorithmic complexity.

**Python** owns the **intelligence layer** — all LLM and RAG logic:
- Document parsing (PDF, DOCX, plain text)
- Text chunking and tokenization
- Embedding generation (Amazon Bedrock Titan)
- Vector similarity search (Qdrant locally, OpenSearch Serverless on AWS)
- Prompt construction
- Answer generation (Amazon Bedrock Claude)

Python is the de facto language of the ML/AI ecosystem. Every embedding library, vector store client, document parser, and LLM SDK has a Python-first implementation. Fighting this by doing RAG in TypeScript adds complexity for no gain.

The boundary is enforced: Python files have no TypeScript neighbours. TypeScript files contain no ML logic. How the two layers integrate in AWS is an open architecture decision — the Python service is independently runnable and testable via `curl` regardless of what calls it.

### The RAG Pipeline

RAG — Retrieval-Augmented Generation — is the pattern of giving a language model factual context at query time rather than relying on its training data. It solves the core limitation of LLMs for enterprise use: they cannot know your private documents.

**Ingestion flow** (write path):

```
Document bytes
  → parse (PDF/DOCX/TXT)
  → chunk (sliding window, 500 tokens, 50 overlap)
  → embed each chunk (Bedrock Titan Text Embeddings v2, 1024 dims)
  → store in vector index with metadata
```

Chunking with overlap preserves context across chunk boundaries — a sentence split across two chunks isn't orphaned in either. The overlap size (50 tokens) is small enough not to waste index space but large enough to bridge most natural paragraph breaks.

**Retrieval + generation flow** (read path):

```
User question
  → embed question (same model as ingestion — critical)
  → kNN vector search: find top-k chunks by cosine similarity
  → build prompt: system prompt + context chunks + question
  → generate answer (Bedrock Claude 3 Sonnet, temperature=0)
  → return answer + source chunks
```

Temperature zero is deliberate. This is a retrieval task, not a creative one. Variance in factual answers is a bug, not a feature. The system prompt instructs the model to stay within the provided context and declare uncertainty rather than hallucinate — this constraint is version-controlled in `specs/prompts.yaml`.

## Getting Started

### Prerequisites

- Node.js 20+
- Python 3.12+
- AWS CLI (for deployment)
- AWS CDK CLI: `npm install -g aws-cdk`

### Local Development (no AWS needed)

The Python service runs in mock mode by default — no Bedrock credentials required. All external calls (embeddings, generation) return stub responses.

```bash
# 1. Install Python dependencies
cd services/rag-service
cp .env.example .env
pip install -r requirements.txt

# 2. Start the service
python main.py
# → Uvicorn running on http://0.0.0.0:8000

# 3. Smoke test
curl http://localhost:8000/health

# 4. Test document upload
curl -X POST http://localhost:8000/upload \
  -F "file=@/path/to/document.pdf"

# 5. Test querying
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"question": "What is this document about?", "top_k": 3}'
```

### Enable Real Bedrock Calls

```bash
# In services/rag-service/.env
MOCK_EMBEDDINGS=false
MOCK_GENERATION=false
AWS_REGION=us-east-1
```

Ensure your AWS credentials have `bedrock:InvokeModel` permission for Titan Embeddings and Claude 3 Sonnet.

### TypeScript Build

```bash
npm install                                    # install all workspaces
npm run build --workspace=packages/shared      # shared types first
npm run build --workspace=apps/api             # Lambda handlers
```

### AWS Deployment

```bash
# One-time bootstrap (per account/region)
cd infra/cdk
npx cdk bootstrap

# Deploy
npm run cdk:deploy

# Outputs include:
#   DocQaStack.ApiUrl            → API Gateway URL
#   DocQaStack.DocumentBucketName → S3 bucket
#   DocQaStack.RagServiceFunctionArn  → Python Lambda ARN
```

## The .claude Folder

The `.claude/` directory is the Claude Code project configuration. It makes every session in this repo immediately context-aware — Claude doesn't need to rediscover the architecture or conventions.

**`CLAUDE.md`** — The project brief loaded at session start. Contains architecture overview, directory purpose, and per-language conventions. Think of it as onboarding documentation written for an AI colleague.

**`settings.json`** — Declares which Bash commands are pre-approved. Without this, Claude prompts for permission before every `npm run` or `curl`. The allowlist covers safe, read-or-build operations; destructive operations (deploy, delete) still require confirmation.

**`commands/`** — Project-specific slash commands, written as markdown. Each file is a task description with optional `$ARGUMENTS`. Claude executes the steps described in the file.

**`agents/`** — Specialized subagents with focused context:
- `rag` — Knows Bedrock API call shapes, the `MOCK_*` pattern, module layout, and how to add new vector store backends
- `cdk-infra` — Knows the current stack topology, CDK v2 patterns, IAM grant methods, and what CfnOutputs to add
- `spec-guardian` — Read-only reviewer that produces a structured `PASS / WARN / FAIL` report against the spec files

## Extending the System

### Adding a Vector Store Backend

1. Create `services/rag-service/retrieval/stores/<name>_store.py` implementing the `search()` interface
2. Register it in `_build_vector_store()` in `retrieval/retriever.py`
3. Add the client to `requirements.txt`
4. Set `VECTOR_STORE=<name>` in `.env`

### Adding a New API Endpoint

1. Define the path, schema, and response in `specs/openapi.yaml`
2. Update `packages/shared/src/types/index.ts` with any new types
3. Create the FastAPI route in `services/rag-service/main.py`
4. Create the Lambda handler in `apps/api/src/handlers/`
5. Wire it to API Gateway in `infra/cdk/lib/stack.ts`

### Switching Language Models

Change `GENERATION_MODEL_ID` in `.env` to any Bedrock-supported Claude model. The `generator.py` uses the Anthropic Messages API format via Bedrock, which is consistent across Claude versions.
